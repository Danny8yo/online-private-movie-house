# 需求分析——管理成员(Manage Members)
## 用例片段： 管理成员——踢出普通用户
范围： 系统验证管理员身份与操作对象权限，弹出确认提示，管理员确认后，系统发送踢出请求至服务器，更新房间成员列表，强制目标用户退出，同时记录操作日志。  
前置条件： 用户已登录。

### 1. 根据UC-06管理成员用例描述提取候选类
- "对应 “参与者” 角色，当管理员进入成员管理界面（观影厅 / 房间类场景映射）时" ——> 管理员（User）  
- “选择目标用户（selectUser (userId)）、点击踢出按钮（clickKick ()）” ——> 成员管理界面（UI）  
- “聊天输入面板 UI 类” ——> 成员管理处理器（Handler） 
- “校验管理员身份（是否为当前房间有权限的管理员）” ——> 权限验证器（PermissionValidator）  
- “响应处理器请求：返回目标用户角色，接收成员列表更新指令” ——> 成员列表管理器（MemberListManager - Entity）  
- “对应 “消息实体”” ——> 操作日志（OperationLog - Entity）  
- “接收处理器的踢出请求, 向服务器发送操作指令，执行实际的踢出动作” ——> 服务器通信器（ServerCommunicator）  

### 2. 健壮性分析
#### 边界类（Boundary）
- MemberManagementUI：负责接收管理员的选择、点击等交互操作，展示确认提示与结果反馈。
#### 控制类（Control）
- MemberManagerHandler：协调踢出用户的完整流程，调用权限验证、服务器通信、日志记录等功能。
- PermissionValidator：专门负责验证管理员身份、操作对象是否为普通用户等权限校验逻辑。
#### 实体类（Entity）
- MemberListManager：维护房间成员列表、记录成员角色（管理员 / 普通用户）及房间状态（正常 / 已解散）。
- OperationLog：封装踢出操作的相关信息（操作人 ID、目标用户 ID、操作时间、操作类型、房间 ID）。 

### 3. 健壮性交互流程
1.	管理员在 MemberManagementUI 选择目标普通用户并点击 “踢出” 按钮。
2.	UI 通知 MemberManagerHandler 处理踢出操作。
3.	MemberManagerHandler 调用 PermissionValidator 验证权限（Subflow S1：验证管理员身份、目标用户为普通用户、当前成员列表状态正常）。
4.	权限验证通过后，MemberManagerHandler 向 MemberListManager 请求当前房间状态（Subflow S2：确认房间为 “正常可用” 状态）。
5.	MemberManagerHandler 调用 ServerCommunicator 发送踢出请求至服务器。
6.	ServerCommunicator 执行踢出操作并返回结果，MemberManagerHandler 调用 MemberListManager 更新成员列表（Subflow S3：移除目标用户 ID、同步成员数量）。
7.	MemberManagerHandler 调用 OperationLog 实体记录踢出操作日志（Subflow S4：记录操作人 ID、目标用户 ID、操作时间、操作类型）。
8.	MemberManagerHandler 将操作结果反馈至 MemberManagementUI，UI 向管理员展示 “踢出成功” 提示
