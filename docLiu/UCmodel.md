# 用例描述模板

## 1. 用例名称 (Use Case Name)

* **名称**：[动词 + 名词短语，使用主动语态，例如“提取现金”]
* **ID**：[例如 UC-01]

## 2. 简要描述 (Brief Description)

* 简明扼要地说明参与者希望通过系统达成的最终目标。不要在此处列出步骤，仅说明发生了什么以及结果是什么。]

## 3. 参与者 (Actors)

* **主要参与者**：[触发用例的人或系统]
* **次要参与者**：[在用例执行过程中与系统交互的其他系统或人员]

## 4. 前置条件 (Preconditions)

* 用例开始前系统必须满足的状态或条件。例如：用户已登录。

## 5. 基本事件流 (Basic Flow)

*[此部分描述一切顺利时的核心流程。遵循以下写作原则：]*

* *使用主动语态和现在时态（例如“系统验证...”而不是“系统将验证...”）。*
* *它是 UI-Free 的：描述系统做什么，而不是界面长什么样（不要写“点击按钮”、“显示对话框”）。 *
* *捕获“屏幕背后”的行为：包括验证逻辑、计算、数据记录等内部行为。 *

1.  **用例启动**：当 [参与者] [执行某个触发动作] 时，用例启动。 
2.  **交互步骤**：系统 [执行某操作/获取某信息]。 
    * *如果涉及复杂验证或业务规则，请详细说明。* 
3.  **子流引用**：执行 [子流名称]（如果有可复用的步骤序列）。 
4.  **交互步骤**：参与者 [提供信息]。
5.  **系统响应与处理**：系统 [验证信息/处理数据/记录日志]。
6.  **用例结束**：当 [达成目标] 时，用例结束。 

## 6. 备选流 (Alternative Flows)

*[描述异常情况、错误处理或可选路径。不要在基本流中混杂这些内容。] *

**A1. [备选流名称，例如：验证失败]**

1.  在基本流步骤 X 中，如果 [发生特定条件]。
2.  系统 [执行错误处理或替代逻辑]。
3.  系统 [终止用例 或 返回基本流步骤 Y]。 [cite: 698]

## 7. 子流 (Subflows)

*[提取复杂的、重复的或逻辑独立的步骤序列，以简化基本流。] *

**S1. [子流名称，例如：验证文件格式]**

1.  系统 [执行子步骤 1]。
2.  系统 [执行子步骤 2]。
3.  回到调用点。

## 8. 后置条件 (Postconditions)

* [用例成功结束后，系统状态的变化。例如：交易记录已保存，库存已扣减。]
